The four dimensions of amenability

1. Oracle clarity
Does a machine-readable ground truth exist? Binary (pass/fail) > numeric target > proxy metric > human judgment required. This is the gating criterion — without an oracle, no loop is possible.
2. Feedback latency
How long after the agent acts can it observe the result? Sub-second (compiler) > seconds (test suite) > minutes (E2E) > hours (deployment) > weeks (user adoption). Latency determines loop speed and how many iterations are feasible.
3. Loop closure
Can the agent observe the result of its own action without human intervention? Closed loop (compiler output → agent) vs. open loop (user feedback → human → agent). Only closed loops can be truly autonomous.
4. Decomposability
Can the task be broken into independently verifiable subtasks? Monolithic "does this architecture work?" is resistant. Decomposed into "does this module pass its contract?" becomes amenable.

The non-obvious structural insights

Verifiability can be engineered
The boundary between amenable and resistant is not fixed. Tests make code verifiable; specs make requirements verifiable; evals make AI outputs verifiable. The move is always: "how do I create an oracle?" not "is an oracle available?"
Future processes cluster high
AI-introduced processes are disproportionately amenable because they were designed with machine-readability as a first-class concern. The AI layer is eating its own interfaces first. CLAUDE.md optimization is a programming loop; "is this a good architecture?" is not.
The two-layer pattern
Most "moderate" processes are actually two processes: a mechanical layer (highly amenable) and a judgment layer (resistant). Code review = pattern matching (perfect) + design judgment (resistant). Refactoring = behavior preservation (perfect) + design quality (resistant). Split them; automate only the former.
The permanently resistant core
"Is this the right thing to build?" will remain resistant indefinitely. The oracle — market adoption — is slow, noisy, and requires reality to run. No amount of AI capability eliminates the need for human judgment at the strategic frontier. This is where senior engineering judgment concentrates.

Implication for software engineering practice

The structurally resistant processes — requirements, product strategy, architectural judgment — are concentrating in value as everything else becomes automatable. This is not a crisis; it is a clarification. The engineering skill that compounds now is: creating verifiability where it didn't exist before — writing the evals, defining the behavioral contracts, specifying what "done" means in machine-readable terms. Every hour you spend writing a good test, a good eval, a good spec, or a good CLAUDE.md entry is multiplied by every future loop iteration that uses it as an oracle. The bottleneck has always been judgment. Now we have empirical proof.
